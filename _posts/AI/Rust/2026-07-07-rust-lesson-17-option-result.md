# 17강: Rust 핵심: Option과 Result 타입

## 1. Option 타입: "값이 있을 수도 있고, 없을 수도 있어!"

먼저 Option부터 가봅시다. 다른 언어에서는 값이 없을 때 null을 사용하죠? 그런데 이 null이 개발자들 사이에서는 "10억 달러짜리 실수"라고 불릴 만큼 악명이 높습니다. 왜냐고요? 값이 있을 줄 알고 썼는데 null이면 프로그램이 그대로 뻗어버리거든요.

Rust는 아주 영리합니다. "값이 없을 수도 있다면, 그걸 명시적으로 표시해서 개발자가 반드시 처리하게 만들자!"라는 생각으로 Option 타입을 만들었습니다.

### Option을 비유하자면? [선물 상자]
Option은 하나의 선물 상자와 같습니다.
- Some(T): 상자를 열었더니 진짜 선물이 들어있는 상태입니다.
- None: 상자를 열었는데 텅 비어있는 상태입니다.

중요한 건, Rust에서는 상자를 열어보기 전까지는 그 안에 내용물을 꺼내 쓸 수 없게 강제한다는 점입니다. 무작정 "선물 꺼내서 써야지!"라고 했다가는 컴파일러가 "야! 상자 안에 선물이 없을 수도 있잖아! 어떻게 처리할 건지 먼저 말해!"라고 소리를 지릅니다.

### [코드 예제 1] Option 타입의 기본과 활용

그럼 실제로 어떻게 쓰는지 코드로 보겠습니다.

```rust
fn main() {
    // 1. Option 타입 변수 선언
    // Some는 값이 있다는 뜻이고, None은 값이 없다는 뜻입니다.
    let some_gift: Option<String> = Some(String::from("최신형 노트북"));
    let no_gift: Option<String> = None;

    println!("첫 번째 상자: {:?}", some_gift);
    println!("두 번째 상자: {:?}", no_gift);

    // 이제 이 상자 안의 값을 어떻게 꺼낼까요? 
    // Rust에서는 그냥 꺼낼 수 없고, 반드시 아래와 같은 방법들을 사용해야 합니다.
}
```

- `Option<String>`: 이 변수는 문자열이 들어있을 수도 있고, 아예 없을 수도 있다는 뜻입니다.
- `Some(String::from("..."))`: 실제로 값이 들어있는 상태를 감싸줍니다.
- `None`: 값이 없는 상태를 명시합니다.
- `{:?}`: Option 타입은 기본 출력 형식이 아니기 때문에 디버그 출력 포맷을 사용해야 합니다.

## 2. Option의 값을 꺼내는 3가지 방법

자, 이제 상자를 열어볼 시간입니다. Rust는 개발자가 실수하지 않도록 다양한 방법을 제공합니다.

### 방법 1: match 문 (가장 꼼꼼한 방법)
모든 경우의 수를 다 따지는 가장 정석적인 방법입니다.

```rust
let gift = Some(String::from("스마트워치"));

match gift {
    Some(item) => println!("와! {}를 받았습니다!", item),
    None => println!("상자가 비어있네요... 슬픕니다."),
}
```
- `Some(item)`: 값이 있을 때 `item`이라는 이름으로 값을 꺼내서 사용합니다.
- `None`: 값이 없을 때 실행할 코드를 작성합니다.
- 장점: 모든 상황을 처리했기 때문에 매우 안전합니다.

### 방법 2: if let 문 (딱 필요한 것만 챙기는 방법)
"값이 없을 때는 딱히 할 말 없으니까, 있을 때만 처리할래!" 할 때 씁니다.

```rust
let gift = Some(String::from("무선 이어폰"));

if let Some(item) = gift {
    println!("럭키! {}를 득템했습니다!", item);
}
```
- `if let Some(item) = gift`: gift가 Some일 때만 중괄호 내부를 실행합니다.
- 장점: 코드가 훨씬 간결해집니다.

### 방법 3: unwrap_or() (기본값을 정해두는 방법)
"값이 없으면 그냥 이걸로 쳐!"라고 기본값을 지정하는 방식입니다.

```rust
let gift: Option<String> = None;
let final_gift = gift.unwrap_or(String::from("사탕 한 알"));

println!("내가 받은 선물은 {}입니다.", final_gift);
```
- `unwrap_or(...)`: 값이 있으면 그 값을 반환하고, None이면 괄호 안의 기본값을 반환합니다.
- 장점: 변수에 값을 할당할 때 매우 유용합니다.

---

> ### 💡 초보자 폭풍 질문!
> **질문: "선생님, 그냥 unwrap() 쓰면 편하던데 왜 이렇게 복잡하게 해요? 그냥 unwrap() 쓰면 안 되나요?"**
>
> **재준봇의 답변:** 
> 오, 아주 날카로운 질문입니다! `unwrap()`은 "상자 안에 무조건 값이 있다고 믿어! 없으면 그냥 프로그램 터뜨려버려!"라고 명령하는 것과 같습니다. 만약 `None`인 상태에서 `unwrap()`을 호출하면 프로그램이 즉시 종료(Panic)됩니다. 실무에서 `unwrap()`을 남발하는 것은 시한폭탄을 설치하는 것과 같아요. 정말로 값이 있을 수밖에 없는 상황이 아니라면, 위에서 배운 안전한 방법들을 사용하는 습관을 들여야 합니다!

## 3. Result 타입: "성공했을 수도 있고, 실패했을 수도 있어!"

Option이 "있냐 없냐"의 문제라면, Result는 "성공했냐 실패했냐"의 문제입니다. 주로 파일을 읽거나 네트워크 통신을 할 때처럼, 외부 요인으로 인해 에러가 발생할 가능성이 높은 작업에 사용합니다.

### Result를 비유하자면? [택배 배송]
Result는 택배 상자와 같습니다.
- Ok(T): 택배가 무사히 도착해서 상품이 들어있는 상태입니다.
- Err(E): 배송 중에 사고가 나서 "배송 실패 사유서"가 들어있는 상태입니다.

Result 타입의 핵심은 단순히 실패했다는 것뿐만 아니라, "왜 실패했는지(에러 내용)"를 함께 담아서 전달할 수 있다는 점입니다.

### [코드 예제 2] Result 타입의 기본 구조

```rust
fn divide(a: f64, b: f64) -> Result<f64, String> {
    if b == 0.0 {
        // 실패했을 때는 Err에 에러 메시지를 담아 보냅니다.
        Err(String::from("0으로 나눌 수 없습니다!"))
    } else {
        // 성공했을 때는 Ok에 결과값을 담아 보냅니다.
        Ok(a / b)
    }
}

fn main() {
    let result = divide(10.0, 0.0);
    println!("결과: {:?}", result);
}
```
- `Result<f64, String>`: 성공하면 `f64` 타입을 주고, 실패하면 `String` 타입의 에러 메시지를 준다는 뜻입니다.
- `Ok(값)`: 작업이 성공했음을 나타냅니다.
- `Err(에러)`: 작업이 실패했음을 나타내며, 실패 원인을 담습니다.

## 4. Result의 값을 처리하는 3가지 방법

Result 역시 Option처럼 안전하게 처리하는 것이 핵심입니다.

### 방법 1: match 문 (정석적인 에러 처리)
에러가 났을 때 어떻게 복구할지 상세하게 적어줄 수 있습니다.

```rust
let result = divide(10.0, 0.0);

match result {
    Ok(value) => println!("계산 성공: {}", value),
    Err(e) => println!("에러 발생: {}", e),
}
```
- 성공과 실패 케이스를 명확히 구분하여 처리합니다.

### 방법 2: ? 연산자 (전문가들의 지름길)
이건 정말 Rust의 꽃입니다. "성공하면 값을 넘기고, 실패하면 여기서 바로 에러를 던져서 상위 함수로 보내버려!"라는 뜻입니다.

```rust
fn calculate_total() -> Result<f64, String> {
    // divide가 Ok를 반환하면 value에 담고, Err를 반환하면 
    // calculate_total 함수 자체가 즉시 Err를 반환하며 종료됩니다.
    let value = divide(10.0, 2.0)?; 
    Ok(value + 10.0)
}
```
- `?` 하나로 수많은 if문과 match문을 대체할 수 있습니다. (단, 반환 타입이 Result여야 사용 가능합니다.)

### 방법 3: expect() (강력한 경고와 함께 종료)
`unwrap()`과 비슷하지만, 프로그램이 터질 때 왜 터졌는지 친절한 메시지를 남기고 싶을 때 씁니다.

```rust
let result = divide(10.0, 0.0);
let value = result.expect("나눗셈 계산 중 치명적인 오류가 발생했습니다!");
```
- `expect("메시지")`: 값이 없으면 패닉을 일으키며 지정한 메시지를 출력합니다. `unwrap()`보다 훨씬 친절하죠.

---

> ### ⚠️ 실무주의보
> **주의사항: 에러를 그냥 무시하지 마세요!**
>
> 초보자분들이 가장 많이 하는 실수가 `let _ = some_function();` 처럼 Result 값을 그냥 버리는 것입니다. 이렇게 하면 에러가 발생해도 프로그램은 아무 말 없이 그냥 지나가게 되고, 나중에 엉뚱한 곳에서 원인을 알 수 없는 버그가 터집니다. Rust가 Result를 반환했다는 것은 "여기서 에러가 날 수 있으니 반드시 처리해!"라고 신호를 보낸 것입니다. 귀찮더라도 반드시 `match`, `if let`, 또는 `?` 연산자를 사용해 처리하세요.

## 5. 마무리: Option vs Result 한눈에 비교하기

마지막으로 헷갈리지 않게 딱 정리해 드릴게요.

| 구분 | Option | Result |
| :--- | :--- | :--- |
| **핵심 개념** | 값의 존재 여부 (있나? 없나?) | 작업의 성공 여부 (됐나? 안 됐나?) |
| **성공 케이스** | `Some(T)` | `Ok(T)` |
| **실패 케이스** | `None` | `Err(E)` |
| **사용 예시** | 리스트에서 특정 인덱스 찾기, 선택적 필드 | 파일 읽기, 네트워크 요청, 나눗셈 |
| **비유** | 선물 상자 (내용물이 있을 수도, 없을 수도) | 택배 배송 (물건이 올 수도, 사고 날 수도) |

여러분, 오늘 배운 Option과 Result는 Rust가 메모리 안전성과 안정성을 보장하는 가장 강력한 도구들입니다. 처음에는 "왜 이렇게까지 해야 해?" 싶겠지만, 익숙해지면 "아, 이래서 Rust가 안전하구나"라고 느끼게 될 거예요.
